# Recurring Transactions

## Purpose

Recurring transactions exist to reduce repeated manual entry for predictable financial activity (e.g., salary, rent, subscriptions). They extend the transaction system rather than replacing it.

## Recurring Rule Concept

A recurring transaction begins as a recurring rule: a template describing a transaction pattern that repeats over time.

- Generated transactions follow normal rules for ownership, accounts, currencies, and required categories (see [Core Concepts](./core-concepts.md) and [Transactions](./transactions.md)).
- Supported types: **Income**, **Expense**.
- **Explicit Exclusion:** `Transfer` transactions cannot be made recurring in the MVP to avoid complex cross-currency historical exchange rate locking.

## Minimum Product Inputs

- Transaction type, amount, currency, account, category
- **Recurrence pattern**: `daily`, `weekly`, `monthly`, `yearly` with a positive interval (e.g., every N days/months).
- **Start date**
- Scheduling uses date-only calendar logic in the app-wide `America/Argentina/Buenos_Aires` timezone.

## User Expectations

**Editing:**
- Editing a rule updates the future behavior only.
- Past generated transactions retain their original values and are not retroactively modified or deleted.

**Pause:**
- Rules can be paused and resumed.
- Missed dates during the paused period are **not backfilled** upon resume.

**Deletion:**
- Rules can be deleted to stop future generation.
- Historical transactions generated by the rule remain intact in the ledger.

## Background Execution Architecture

Since the MVP deployment topology lacks a dedicated task queue, recurring transactions run on an in-process cron scheduler inside the `api` service.

See [Background Jobs and Concurrency](../platform/background-jobs.md) for architectural details on execution, missed tick catch-up, and future concurrency handling.
